# ВВЕДЕНИЕ

Современный мир диктует новые требования к образованию и оценке знаний. Пандемия 2020 года и последовавший за ней переход к дистанционному и гибридному формату обучения коренным образом изменили подход педагогов, преподавателей и HR-специалистов к проверке знаний и компетенций. Уже сегодня дистанционное тестирование стало неотъемлемой частью учебного процесса в школах, колледжах и вузах, а также инструментом оценки персонала в организациях.

Вместе с тем существующие решения — Google Forms, Яндекс.Формы, Kahoot и другие — имеют ряд существенных ограничений: зависимость от корпоративных аккаунтов, ограниченный бесплатный функционал, сложность настройки, необходимость регистрации для участников, а также ограниченные возможности для анализа результатов. Кроме того, в условиях работы на территории Российской Федерации ряд зарубежных сервисов недоступен без дополнительного программного обеспечения, что существенно затрудняет их использование.

В данной дипломной работе описана разработка оригинального веб-приложения «Конструктор тестов» (take-test.ru) — инструмента для создания, распространения и анализа тестирований, лишённого перечисленных недостатков. Особенностью приложения является реализация по технологии **Progressive Web App (PWA)**, позволяющей установить его на любое устройство — смартфон, планшет или компьютер — без App Store и Google Play, а также обеспечивающей частичную работу без подключения к интернету.

При разработке применялась методология **Vibe Coding** — подход, при котором разработчик формулирует задачи на естественном языке, а искусственный интеллект (в данном случае Claude от Anthropic) генерирует, анализирует и дорабатывает программный код. Такой подход позволяет значительно ускорить разработку, однако требует от программиста понимания архитектурных решений, умения ставить корректные задачи и проверять результат.

**Цель дипломной работы** — спроектировать и реализовать полнофункциональное кроссплатформенное веб-приложение для создания и прохождения тестов, доступное на любых устройствах, с собственной системой авторизации, несколькими форматами вопросов и механизмом анализа результатов.

# 1. АНАЛИЗ ПРЕДМЕТНОЙ ОБЛАСТИ ДЛЯ РАЗРАБОТКИ ПРИЛОЖЕНИЯ

## 1.1. Описание предметной области и обоснование актуальности разработки

Под тестированием в образовательном и корпоративном контексте понимается процедура проверки знаний, умений и компетенций с помощью стандартизированных заданий — тестов. Тест состоит из набора вопросов (items), каждый из которых предполагает один или несколько вариантов ответа, либо свободный ввод. По результатам прохождения формируется числовая оценка — балл.

Дистанционное тестирование приобрело массовый характер по ряду причин:

- **Масштабируемость.** Один тест может пройти одновременно несколько тысяч человек в разных городах без организации физического присутствия.
- **Скорость обработки результатов.** Автоматическая проверка позволяет получить результат мгновенно.
- **Снижение субъективности.** Компьютерная проверка исключает человеческий фактор при оценивании формальных ответов.
- **Экономия ресурсов.** Не требуются бумага, аудитории, организаторы.

По данным исследования компании Statista, объём глобального рынка систем управления обучением (LMS) в 2023 году составил около 21 млрд долларов, а к 2028 году прогнозируется рост до 47 млрд долларов [1]. Это подтверждает высокий и растущий спрос на инструменты дистанционного обучения и тестирования.

В Российской Федерации распространение дистанционного образования дополнительно стимулируется рядом факторов:

1. **Географическая протяжённость страны** — студенты и сотрудники нередко находятся в разных часовых поясах.
2. **Действие Федерального закона № 273-ФЗ** «Об образовании в Российской Федерации», допускающего применение электронного обучения и дистанционных технологий (статья 16).
3. **Активное развитие мобильного интернета** — по данным Минцифры России, в 2024 году охват сетями 4G составил более 85% населённых пунктов страны.

Тем не менее анализ рынка показывает: большинство доступных сервисов тестирования либо требуют регистрации участников, либо ограничивают количество вопросов/участников в бесплатной версии, либо недоступны на территории России без VPN. Разработка собственного приложения позволяет устранить эти ограничения и адаптировать инструмент под конкретные потребности пользователей.

## 1.4. Обоснование выбора технологии PWA для кроссплатформенной разработки

**Progressive Web App (PWA)** — это веб-приложение, которое использует возможности современных браузеров для имитации поведения нативного приложения. Концепция введена Google в 2015 году. Ключевые компоненты PWA:

1. **Web App Manifest** — JSON-файл с описанием иконок, названия, цветовой схемы и режима отображения.
2. **Service Worker** — скрипт, работающий в фоновом потоке браузера, перехватывающий сетевые запросы и управляющий кэшем.
3. **HTTPS** — обязательное условие работы Service Worker.

**Таблица 1.5 — Сравнение PWA с нативными приложениями и обычными сайтами**

| Характеристика | Нативное приложение | Обычный сайт | PWA |
|---|---|---|---|
| Установка с магазина | Обязательна | Нет | Нет |
| Работа офлайн | Да | Нет | Да |
| Доступ к API устройства | Полный | Ограниченный | Частичный |
| Обновление | Вручную/автоматически | Автоматически | Автоматически |
| Один кодовая база | Нет (iOS + Android) | Да | Да |
| Затраты на разработку | Высокие | Средние | Средние |
| Публикация в App Store | Обязательна | Нет | Нет (опционально) |
| Доступность | Через установку | Через браузер | Через браузер + установка |

Для разрабатываемого приложения PWA стала оптимальным выбором по следующим причинам:

- **Единая кодовая база** — нет необходимости разрабатывать отдельные приложения для iOS и Android.
- **Отсутствие модерации App Store** — публикация в магазинах приложений требует оплаты (99 $/год для Apple Developer), проверки до 7 дней и может быть отклонена.
- **Мгновенное обновление** — пользователи всегда получают актуальную версию без ручного обновления.
- **Работа без интернета** — Service Worker кэширует ресурсы, что критично при нестабильном соединении.
- **Низкий порог входа** — участник теста просто переходит по ссылке, не скачивая ничего.

**Таблица 1.6 — Поддержка PWA в браузерах (2024)**

| Браузер | Платформа | Установка PWA | Service Worker | Push-уведомления |
|---|---|---|---|---|
| Chrome 120+ | Windows/Android | Да | Да | Да |
| Safari 17+ | iOS/macOS | Да | Да | Да (macOS) |
| Firefox 120+ | Desktop | Нет | Да | Да |
| Edge 120+ | Windows | Да | Да | Да |
| Samsung Internet | Android | Да | Да | Да |

Таким образом, PWA поддерживается в 95%+ используемых браузеров, что обеспечивает широкий охват аудитории.

## 1.5. Обоснование выбора языка программирования и инструментов разработки

При выборе технологического стека оценивались следующие критерии: скорость разработки, доступность документации на русском языке, наличие готовых библиотек, возможность бесплатного хостинга, сообщество и поддержка.

### Фронтенд: React + Vite

**React** — JavaScript-библиотека для построения пользовательских интерфейсов, разработанная Facebook (Meta). Использует концепцию компонентов — самостоятельных блоков UI с собственным состоянием. На момент разработки React является наиболее популярным фронтенд-инструментом в мире.

**Таблица 1.7 — Сравнение фронтенд-фреймворков**

| Параметр | React 19 | Vue 3 | Angular 17 | Svelte 5 |
|---|---|---|---|---|
| Звёзд на GitHub | 230k+ | 47k+ | 95k+ | 80k+ |
| Размер bundle (gzip) | ~45 КБ | ~23 КБ | ~150 КБ | ~15 КБ |
| Кривая обучения | Средняя | Низкая | Высокая | Низкая |
| Экосистема библиотек | Огромная | Большая | Большая | Малая |
| Поддержка AI-инструментов | Отличная | Хорошая | Хорошая | Удовлетворительная |
| Используется в проекте | **Да** | Нет | Нет | Нет |

**Vite** — современный инструмент сборки, заменяющий устаревший Webpack. Использует ES Modules в режиме разработки — перезагрузка изменений происходит мгновенно (Hot Module Replacement). Сборка для продакшена основана на Rollup.

### Стилизация: Tailwind CSS

**Tailwind CSS** — утилитарный CSS-фреймворк: вместо написания CSS-правил разработчик применяет готовые классы прямо в разметке. Например, `className="bg-indigo-600 text-white rounded-lg px-4 py-2"` задаёт синий фон, белый текст, скруглённые углы и отступы — без единой строки CSS.

Преимущество для данного проекта: полная свобода в дизайне без конфликтов стилей, встроенная адаптивность (`sm:`, `md:`, `lg:` префиксы), минимальный итоговый CSS (Tailwind 4 удаляет неиспользуемые классы при сборке).

### Бэкенд: Python + FastAPI

**Python** выбран как язык бэкенда по следующим причинам:
- Широко изучается в российских учебных заведениях
- Богатая экосистема библиотек (пакетный менеджер pip)
- Отличная поддержка в AI-инструментах (большинство примеров — на Python)
- Нативная интеграция с Pydantic для валидации данных

**FastAPI** — современный высокопроизводительный Python-фреймворк для создания REST API. Особенности:
- Автоматическая генерация документации (Swagger UI по адресу `/docs`)
- Встроенная валидация через Pydantic
- Асинхронная поддержка (async/await)

### Хранение данных: SQLite + SQLAlchemy

**SQLite** — встраиваемая реляционная база данных. Файл базы данных (`tests.db`) хранится прямо на сервере. Для данного проекта (один сервер, небольшая нагрузка) SQLite является оптимальным решением, не требующим отдельного сервера БД.

### Угрозы и меры защиты

**Таблица 2.1 — Матрица угроз и контрмер**

| Угроза | Описание | Контрмера |

### Двухшаговая верификация

Регистрация разбита на два шага намеренно:

**Шаг 1** — проверка данных и отправка кода:
- Валидация email (Pydantic `EmailStr`)
- Проверка формата логина (regex `^[a-zA-Z0-9_]+$`)
- Проверка длины пароля (≥ 6 символов)
- Проверка уникальности email и логина
- Генерация кода: `secrets.randbelow(900000) + 100000` (6 цифр, криптографически безопасный)
- Срок действия: `datetime.utcnow() + timedelta(minutes=10)`

**Шаг 2** — подтверждение кода:
- Поиск записи в `email_codes` по email + purpose = "register"
- Сравнение введённого кода
- Проверка `expires_at < datetime.utcnow()`
- Активация пользователя: `user.is_verified = True`
- Удаление использованных кодов из таблицы

### Хранение паролей

```python
from passlib.context import CryptContext

pwd_context = CryptContext(schemes=["bcrypt"], deprecated="auto")

def hash_password(password: str) -> str:
    return pwd_context.hash(password)

def verify_password(plain: str, hashed: str) -> bool:
    return pwd_context.verify(plain, hashed)
```

bcrypt автоматически добавляет случайную «соль» (salt) к каждому хешу, что делает невозможным использование rainbow-таблиц.

### Валидация входящих данных с помощью Pydantic v2

Pydantic v2 автоматически проверяет входящие данные и возвращает читаемые ошибки при несоответствии:

```python
# schemas.py
from pydantic import BaseModel, EmailStr, field_validator
import re


class RegisterStart(BaseModel):
    email: EmailStr       # Автоматическая валидация email-формата
    login: str
    password: str

    @field_validator('login')
    @classmethod
    def validate_login(cls, v: str) -> str:
        v = v.strip()
        if len(v) < 3:
            raise ValueError('Логин минимум 3 символа')
        if len(v) > 30:
            raise ValueError('Логин не более 30 символов')
        if not re.match(r'^[a-zA-Z0-9_]+$', v):
            raise ValueError(
                'Логин может содержать только латинские буквы, цифры и знак _'
            )
        return v

    @field_validator('password')
    @classmethod
    def validate_password(cls, v: str) -> str:
        if len(v) < 6:
            raise ValueError('Пароль минимум 6 символов')
        return v
```

### Критически важная реализация проверки дубликатов

Один из нетривиальных аспектов реализации — корректная проверка дубликатов при регистрации. Наивная реализация с одним запросом `OR` приводит к ошибке:

```python
# НЕПРАВИЛЬНО — возвращает только одного пользователя,
# даже если оба условия истинны
user = db.query(User).filter(
    (User.email == email) | (User.login == login)
).first()
```

Если в БД есть верифицированный пользователь с таким email И другой пользователь с таким login, запрос вернёт только первого. Удаление этого «первого» нарушит уникальное ограничение по второму.

**Правильная реализация — два отдельных запроса:**

```python
# main.py — endpoint /auth/register/start
@app.post("/auth/register/start")
def register_start(data: schemas.RegisterStart, db: Session = Depends(get_db)):
    email = data.email.strip().lower()
    login = data.login.strip()

    # Два раздельных запроса — находим оба конфликта
    email_user = db.query(models.User).filter(
        models.User.email == email
    ).first()
    login_user = db.query(models.User).filter(
        models.User.login == login
    ).first()

    # Если запись верифицирована — отказ с понятным сообщением
    if email_user and email_user.is_verified:
        raise HTTPException(
            status_code=400,
            detail="Эта почта уже зарегистрирована"
        )
    if login_user and login_user.is_verified:
        raise HTTPException(
            status_code=400,
            detail="Этот логин уже занят"
        )

    # Неверифицированные старые записи удаляем
    # (пользователь мог начать регистрацию но не подтвердить)
    if email_user:
        db.delete(email_user)
    if login_user and login_user is not email_user:
        db.delete(login_user)
    db.commit()

    # Создаём нового пользователя
    user = models.User(
        email=email,
        login=login,
        password_hash=hash_password(data.password),
        is_verified=False,
    )
    db.add(user)

    # Генерируем и сохраняем код подтверждения
    code = generate_code()
    db.add(models.EmailCode(
        email=email,
        code=code,
        purpose="register",
        expires_at=code_expiry(),
    ))
    db.commit()

    send_code_email(email, code, "register")
    return {"sent": True}
```

### Отправка email через Resend

**Почему не SMTP?** Бесплатный тариф платформы Render блокирует исходящие TCP-соединения на портах 465 и 587 (стандартные SMTP-порты). Это делает отправку через Gmail, Яндекс и другие почтовые серверы невозможной. Решением стал **Resend** — сервис для разработчиков, предоставляющий HTTP API для отправки писем через порт 443 (HTTPS), который не блокируется.

```python
# mailer.py
import os
import requests as http

RESEND_API_KEY = os.environ.get("RESEND_API_KEY", "")
FROM_EMAIL = os.environ.get("FROM_EMAIL", "noreply@take-test.ru")


def send_code_email(to_email: str, code_or_text: str, purpose: str):
    if purpose == "register":
        subject = "Подтверждение регистрации — Конструктор тестов"
        body = (
            f"Здравствуйте!\n\n"
            f"Ваш код подтверждения: {code_or_text}\n\n"
            f"Код действует 10 минут.\n"
            f"Если вы не регистрировались, проигнорируйте это письмо."
        )
    elif purpose == "reset":
        subject = "Восстановление пароля — Конструктор тестов"
        body = (
            f"Здравствуйте!\n\n"
            f"Код для сброса пароля: {code_or_text}\n\n"
            f"Код действует 10 минут."
        )
    else:
        subject = "Уведомление — Конструктор тестов"
        body = code_or_text

    if not RESEND_API_KEY:
        # В режиме разработки просто выводим код в консоль
        print(f"[DEV EMAIL → {to_email}] {subject}\n{body}")
        return

    try:
        resp = http.post(
            "https://api.resend.com/emails",
            headers={
                "Authorization": f"Bearer {RESEND_API_KEY}",
                "Content-Type": "application/json",
            },
            json={
                "from": f"Конструктор тестов <{FROM_EMAIL}>",
                "to": [to_email],
                "subject": subject,
                "text": body,
            },
            timeout=10,
        )
        if resp.status_code in (200, 201):
            print(f"[EMAIL OK] Письмо отправлено на {to_email}")
        else:
            print(f"[EMAIL ERROR] {resp.status_code}: {resp.text}")
    except Exception as e:
        print(f"[EMAIL ERROR] {e}")
```

Важная деталь: если `RESEND_API_KEY` не задан (локальная разработка), функция просто выводит код в консоль. Это позволяет тестировать без реальной отправки писем.

```javascript
// Универсальная функция извлечения ошибки из ответа FastAPI
// FastAPI/Pydantic v2 возвращает detail как строку или массив объектов
function extractError(errData, fallback) {
    if (!errData) return fallback;
    if (typeof errData.detail === 'string') return errData.detail;
    if (Array.isArray(errData.detail) && errData.detail.length > 0) {
        const msg = errData.detail[0].msg || '';
        if (msg.toLowerCase().includes('email')) return 'Некорректный формат почты';
        return msg.replace(/^Value error,\s*/i, '') || fallback;
    }
    return fallback;
}

// Шаг 1 регистрации
export async function registerStart(email, login, password) {
    const res = await fetch(`${API_URL}/auth/register/start`, {
        method: 'POST',
        headers: { 'Content-Type': 'application/json' },
        body: JSON.stringify({ email, login, password }),
    });
    if (!res.ok) {
        const err = await res.json().catch(() => ({}));
        throw new Error(extractError(err, 'Ошибка регистрации'));
    }
    return res.json();
}
```

### Реализация вопросов типа «Сопоставление»

Это наиболее технически интересный компонент приложения. Пользователь должен соотнести элементы левого столбца с элементами правого. Правые элементы перемешиваются при каждом прохождении.

Для обозначения пар используются номера (1, 2, 3...) для левого столбца и буквы (A, B, C...) для правого, что значительно упрощает отслеживание выбора:

```jsx
// src/App.jsx — компонент MatchTaking
function MatchTaking({ question, matches, onMatch }) {
    const LETTERS = ['A', 'B', 'C', 'D', 'E', 'F', 'G', 'H', 'I', 'J'];
    const [activeLeftId, setActiveLeftId] = useState(null);

    // Перемешиваем правые элементы один раз при монтировании
    const [shuffledRights] = useState(() => {
        const rights = question.pairs.map((p) => p.right
            ? { id: p.id, ...p.right }
            : { id: p.id, text: '' }
        );
        // Алгоритм Фишера-Йетса
        for (let i = rights.length - 1; i > 0; i--) {
            const j = Math.floor(Math.random() * (i + 1));
            [rights[i], rights[j]] = [rights[j], rights[i]];
        }
        return rights;
    });

    // Сопоставление rightId → буква
    const rightLabel = {};
    shuffledRights.forEach((r, i) => {
        rightLabel[r.id] = LETTERS[i] || String(i + 1);
    });

    const handleLeftClick = (leftId) => {
        // Повторный клик — сброс выделения
        setActiveLeftId((prev) => (prev === leftId ? null : leftId));
    };

    const handleRightClick = (rightId) => {
        if (activeLeftId == null) return;
        onMatch(activeLeftId, rightId);   // Передаём пару наверх
        setActiveLeftId(null);
    };

    // Показываем подсказку в зависимости от состояния
    const activeNum = activeLeftId != null
        ? pairs.findIndex((p) => p.id === activeLeftId) + 1
        : null;

    return (
        <div>
            {/* Подсказка */}
            <div className={`text-xs text-center py-1.5 px-3 rounded-lg mb-3 font-medium ${
                activeLeftId != null ? 'bg-indigo-50 text-indigo-700'
                : allMatched ? 'bg-emerald-50 text-emerald-700'
                : 'bg-gray-50 text-gray-500'
            }`}>
                {activeLeftId != null
                    ? `Выбрано: ${activeNum} — выберите букву в правом столбце`
                    : allMatched
                    ? 'Все элементы сопоставлены. Нажмите чтобы изменить.'
                    : 'Нажмите на элемент слева, затем на соответствующий справа'}
            </div>
            {/* Левый и правый столбцы */}
        </div>
    );
}
```

**Алгоритм Фишера-Йетса** (используемый для перемешивания) гарантирует равномерное распределение: каждая перестановка элементов равновероятна. Без него был бы использован `sort(() => Math.random() - 0.5)`, который даёт неравномерное распределение.

### Подсчёт результатов для разных типов вопросов

```javascript
// Логика подсчёта в handleSubmit
const result = questions.map((q) => {
    if (q.format === 'match') {
        // Сопоставление: все пары должны быть верными
        const userMatches = answers[q.id]?.matches || {};
        const allCorrect = q.pairs.every(
            (p) => userMatches[p.id] === p.id
        );
        return { questionId: q.id, correct: allCorrect, score: allCorrect ? 1 : 0 };

    } else if (q.format === 'options') {
        // Варианты: проверяем совпадение выбранных
        const selected = answers[q.id]?.selected || [];
        const correctIds = q.options
            .filter((o) => o.correct)
            .map((o) => o.id);
        const isCorrect =
            selected.length === correctIds.length &&
            selected.every((id) => correctIds.includes(id));
        return { questionId: q.id, correct: isCorrect, score: isCorrect ? 1 : 0 };

    } else if (q.format === 'text') {
        // Открытый ответ: нечувствительное к регистру сравнение
        const typed = (answers[q.id]?.text || '').trim().toLowerCase();
        const expected = (q.correctText || '').trim().toLowerCase();
        const isCorrect = expected !== '' && typed === expected;
        return { questionId: q.id, correct: isCorrect, score: isCorrect ? 1 : 0 };
    }
});
```

| `www` | CNAME | cname.vercel-dns.com | Сайт с www → Vercel |
| `api` | CNAME | (render-server).onrender.com | API → Render |
| `resend._domainkey` | TXT | p=MIGfMA0GCSq... (DKIM-ключ) | Цифровая подпись писем |
| `@` | MX | feedback-smtp.us-east-1.amazonses.com | Почтовый шлюз Resend |
| `@` | TXT | v=spf1 include:amazonses.com ~all | SPF: авторизация отправителя |
| `_dmarc` | TXT | v=DMARC1; p=none; | DMARC: политика обработки |

**SPF, DKIM, DMARC** — три стандарта аутентификации email:
- **SPF** (Sender Policy Framework) — указывает, какие серверы вправе отправлять письма от имени домена
- **DKIM** (DomainKeys Identified Mail) — цифровая подпись каждого письма, подтверждающая подлинность отправителя
- **DMARC** — политика обработки писем, не прошедших SPF/DKIM (игнорировать / помещать в спам / отклонять)

Без этих записей письма с `@take-test.ru` попадали бы в папку «Спам» или отклонялись почтовыми серверами.

## 3.4. Применение методологии Vibe Coding при разработке

### Что такое Vibe Coding

**Vibe Coding** — современный подход к разработке программного обеспечения, при котором разработчик формулирует задачу на естественном языке (включая русский), а нейросетевой инструмент (в данном случае **Claude** от Anthropic) генерирует программный код, объясняет его работу и вносит исправления по запросу.

Термин введён Андреем Карпатым (один из основателей OpenAI) в 2025 году и описывает смещение роли программиста: от написания каждой строки кода вручную к грамотной постановке задач и контролю результата.

### Как это применялось в проекте

В ходе разработки данного приложения Claude использовался как:

1. **Архитектурный советник** — помогал выбрать технологии, обосновать структуру проекта.
2. **Генератор кода** — писал компоненты React, FastAPI-маршруты, SQL-модели.
3. **Отладчик** — анализировал ошибки из логов и предлагал исправления.
4. **Инфраструктурный помощник** — давал пошаговые инструкции по настройке DNS, Vercel, Render, Resend.
5. **Ревьюер** — проверял код на баги до его развёртывания.

**Таблица 3.6 — Задачи, решённые с помощью Vibe Coding**

| Задача | Сложность | Результат |
|---|---|---|
| Обнаружение бага с дубликатами при регистрации | Высокая | Исправлен критический баг с `OR`-запросом |
| Реализация типа вопроса «Сопоставление» | Высокая | Полноценный компонент с перемешиванием и буквами |
| Переход с SMTP на HTTP API | Средняя | Переписан mailer.py под Resend |
| Настройка PWA (Service Worker, манифест) | Высокая | Полная офлайн-поддержка через Workbox |
| DNS-конфигурация (SPF, DKIM, DMARC) | Высокая | Все записи верифицированы |
| Парсинг ошибок Pydantic v2 | Средняя | Написана функция extractError() |

### Ограничения и ответственность разработчика

Vibe Coding не означает отсутствие знаний. Разработчик несёт ответственность за:
- **Понимание архитектуры** — AI может предложить плохое решение, если задача сформулирована некорректно
- **Проверку кода** — сгенерированный код нужно читать и понимать перед запуском
- **Безопасность** — AI не всегда учитывает специфические угрозы конкретного проекта
- **Постановку задач** — качество кода напрямую зависит от качества описания задачи

В данном проекте разработчик самостоятельно принимал все архитектурные решения, формулировал требования, тестировал результат и вносил корректировки.
